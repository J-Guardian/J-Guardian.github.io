Libvirt 未授权访问（16509端口）

Libvirt 默认端口（16509/TCP）的用途及安全解析

1. 端口核心用途

16509/TCP 是 libvirt 守护进程（libvirtd）的默认监听端口，主要用于：

  • 远程虚拟机管理：允许通过网络调用 libvirt API（如 virsh、OpenStack、云管平台）管理 KVM/QEMU/Xen 等虚拟化资源。
  • 跨主机协作：支持虚拟机Live Migration）、集群监控等高阶功能。

2.测试命令和批量测试

virsh -c qemu+tcp://localhost/system list –all

VNC未授权漏洞访问漏洞（590x端口）

  • VNC = 图形化远程控制协议，适用于运维、虚拟机和跨设备访问。
  • 优势：跨平台、兼容性强；劣势：性能较低、需加固安全。
  • 虚拟化场景：KVM/QEMU依赖VNC提供虚拟机控制台，需注意端口暴露风险。

1. 什么是VNC？

VNC（虚拟网络计算）是一种图形化远程桌面协议，允许用户通过网络连接到另一台计算机的桌面环境，并实时交互操作（就像坐在那台电脑前一样）。
​​核心特点- ​​跨平台​**​：支持 Windows、Linux、macOS 甚至树莓派。

  • 开源实现：如 RealVNC、TightVNC、TigerV- 协议基于RFB（Remote Framebuffer Protocol），传输屏幕像素和输入事件（键盘/鼠标）。

2. VNC的核心用途

场景 说明
远程办公/技术支持 管理员远程维护服务器或帮助同事解决电脑问题。
虚拟化管理 连接KVM/QEMU虚拟机的图形控制台（如 virt-manager 默认用VNC显示虚拟机画面）。
嵌入式设备调试 操作无显示器的设备（如树莓派、工控机）。
跨平台访问 从手机/平板访问电脑桌面（通过VNC客户端App）。

3. VNC vs 其他远程协议

协议 优势 劣势
VNC 跨平台、开源、支持裸机/虚拟机 传输效率低（高延迟）、无硬件加速
RDP 微软优化、流畅度高（Windows首选） 对非Windows支持有限
SSH -X 加密传输、适合Linux单应用转发 复杂、不支持完整桌面
TeamViewer 易用、穿透NAT 商业软件、隐私争议

4. VNC如何工作？

  1. 服务端（VNC Server）

    • 运行在被控计算机上，持续捕获屏幕画面。
    • 监听默认端口 5900+N（N为显示编号，如 :0=5900, :1=5901）。

  2. 客户端（VNC Viewer）

    • 输入服务端IP和端口发起连接。
    • 接收画面并发送键盘/鼠标指令。

kubelet API 未授权访问漏洞（10255端口）

1. Kubelet 是什么？

Kubelet 是 Kubernetes 集群中的核心组件，运行在每个工作节点（Worker Node）上，负责：

  • 管理Pod生命周期：创建、启动、停止、删除容器（通过容器运行时如 Docker/containerd）。
  • 监控资源：汇报节点和 Pod 的 CPU/内存/磁盘状态给 API Server。
  • 执行指令：响应 API Server 的调度命令（如部署新 Pod）。
  • 暴露API：提供本地 HTTP/HTTPS 接口（默认端口 10250/10255）用于调试和监控。

kubernetes(k8s)和KVM：

技术 定位 核心功能 抽象层级
KVM 虚拟化技术 在物理服务器上创建完整的虚拟机（VM） 硬件级虚拟化（IaaS）
Kubernetes 容器编排平台 管理容器化应用（如Docker），调度微服务 应用级编排（PaaS）

  • KVM 先虚拟化物理服务器，创建多个虚拟机（Worker Nodes），然后在这些虚拟机上部署 Kubernetes 集群。
    优势：

    • 资源隔离更彻底（每个K8s节点是独立VM）。
    • 兼容传统虚拟化运维体系（如OpenStack）。

2. Kubelet API 的用途

API 端口 协议 用途 默认认证
10250 HTTPS 读写接口（管理 Pod、日志、执行命令等） 需 TLS 客户端证书认证
10255 HTTP 只读接口（获取节点状态、Pod 列表等，Kubernetes 1.10+ 已弃用） 无认证（高危！）

3. Kubelet 未授权访问漏洞的形成

当 Kubelet API 的认证配置不当（尤其是 10250 端口），攻击者可能直接访问并控制节点上的容器。常见原因包括：

(1) 错误配置 --anonymous-auth=true

  • 在 Kubelet 启动参数中，若启用匿名访问：

    攻击者可绕过认证调用 API。

(2) 未启用 TLS 或证书验证不严

  • 若 --tls-cert-file 和 --tls-private-key-file 未正确配置，或客户端证书检查不严格（--client-ca-file 未设置），攻击者可能伪造身份。

(3) 遗留的只读端口（10255）开放

  • Kubernetes 1.10 后已弃用 10255（HTTP 明文传输），但若未关闭，攻击者可通过此端口获取敏感信息。

5. 检测方法

（1）扫描开放端口

（2）手动测试未授权访问

  • 返回 200 并显示数据 → 存在未授权访问。
  • 返回 403 Forbidden → 已启用认证。
